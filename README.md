Система управления мероприятиями (Event Management)
Данный проект представляет собой полнофункциональную платформу для управления мероприятиями, включающую backend на Spring Boot, реляционную базу данных, REST API, SPA-клиент, а также автоматизацию развертывания с помощью Docker и CI/CD. Реализованы все ключевые требования лабораторных работ: от базового CRUD до сложных запросов, асинхронных операций, кэширования, тестирования и деплоя.

Модель данных
Проект включает пять основных сущностей:

Event (мероприятие) – название, дата, место.

Session (сессия/доклад) – тема, спикер (связь многие ко многим).

Speaker (спикер) – имя, биография.

Attendee (участник) – имя, email.

Ticket (билет) – тип, цена, привязка к участнику и мероприятию.

Связи:

Event (1) ←→ Session (M) – одно мероприятие содержит много сессий (OneToMany).

Speaker (M) ←→ Session (M) – спикер может выступать на нескольких сессиях, сессия может иметь несколько спикеров (ManyToMany).

Attendee (1) ←→ Ticket (M) – участник может приобрести несколько билетов (OneToMany).

Ticket (M) ←→ Session (M) – реализовано через дополнительную связь (регистрация на сессии), билет привязан к мероприятию, но может давать доступ к нескольким сессиям (многие ко многим).

Функциональные возможности
Проект последовательно реализует все требования лабораторных работ:

Этап 1: Базовое CRUD приложение
Создано Spring Boot приложение.

Реализован REST API для сущности Event.

Endpoint с @PathVariable и @RequestParam.

Слои: Controller → Service → Repository.

DTO и маппинг (MapStruct).

Настроен Checkstyle (Google Java Style).

Этап 2: Работа с БД и связями
Подключена PostgreSQL.

Реализованы все 5 сущностей со связями (OneToMany, ManyToMany).

CRUD для всех сущностей.

Выбраны и обоснованы CascadeType и FetchType.

Продемонстрирована проблема N+1 и её решение через @EntityGraph и fetch join.

Реализован метод массового сохранения данных с демонстрацией транзакционности (@Transactional).

Подготовлена ER-диаграмма.

Этап 3: Сложные запросы и кэширование
Сложный GET-запрос с фильтрацией по вложенной сущности через @Query (JPQL).

Аналогичный запрос через native query.

Пагинация (Pageable).

Реализован in-memory индекс на основе HashMap для кэширования результатов запросов по составному ключу (параметры запроса). Корректная работа за счёт переопределения equals и hashCode.

Инвалидация кэша при изменении данных.

Этап 4: Обработка ошибок, логирование, AOP, Swagger
Глобальный обработчик ошибок (@ControllerAdvice) с единым форматом ответа.

Валидация входных DTO (@Valid).

Логирование через Logback (уровни INFO, DEBUG, ротация файлов по размеру и времени).

Аспект (AOP) для логирования времени выполнения методов сервисов.

Подключен Swagger (OpenAPI) с описанием всех endpoint и моделей.

Этап 5: Bulk-операции, Stream API, транзакции, тесты
Реализована bulk-операция (массовая регистрация участников на мероприятие).

В сервисном слое активно используется Stream API и Optional.

Показана разница поведения с @Transactional и без (частичное сохранение vs полный откат при ошибке).

Написаны unit-тесты для сервисов с использованием Mockito.

Этап 6: Асинхронность, потокобезопасность, нагрузочное тестирование
Реализована асинхронная рассылка уведомлений участникам перед мероприятием (@Async, CompletableFuture).

Эндпоинт для получения статуса задачи.

Потокобезопасный счётчик обработанных уведомлений (AtomicInteger).

Демонстрация race condition и его устранение (сравнение с synchronized).

Нагрузочное тестирование с помощью JMeter (скриншоты результатов в docs/jmeter).

Этап 7: SPA-клиент
Разработан клиент на React.

Клиент взаимодействует с API, отображает связи OneToMany и ManyToMany.

Реализованы CRUD операции и фильтрация.

Этап 8: Docker, CI/CD, деплой
Подготовлен Dockerfile для сборки образа приложения.

docker-compose.yml для запуска приложения + PostgreSQL.

Использование переменных окружения для конфигурации.

Приложение развернуто на бесплатном хостинге (Railway/Render).

Настроен GitHub Actions: сборка, тесты, деплой, healthcheck.

Запуск приложения
Локальный запуск
Требования:

Java 17+

Maven 3.8+

PostgreSQL (можно использовать Docker)

Клонировать репозиторий:

bash
git clone https://github.com/yourusername/event-management.git
cd event-management
Настроить подключение к БД в application-local.properties (или через переменные окружения):

properties
spring.datasource.url=jdbc:postgresql://localhost:5432/eventdb
spring.datasource.username=postgres
spring.datasource.password=postgres
Собрать и запустить:

bash
./mvnw clean spring-boot:run -Dspring-boot.run.profiles=local
Приложение будет доступно по адресу: http://localhost:8080

Запуск через Docker
Убедитесь, что установлены Docker и Docker Compose.

В корне проекта выполнить:

bash
docker-compose up -d
Приложение запустится вместе с БД. Порт: 8080.

Переменные окружения заданы в docker-compose.yml и могут быть переопределены.

API документация
После запуска Swagger UI доступен по адресу:
http://localhost:8080/swagger-ui.html

Основные endpoint:

GET /api/events – список мероприятий (с фильтрацией по дате, месту, пагинация)

GET /api/events/{id} – мероприятие по ID

POST /api/events – создать мероприятие

PUT /api/events/{id} – обновить мероприятие

DELETE /api/events/{id} – удалить мероприятие

Аналогичные CRUD для Session, Speaker, Attendee, Ticket.

Сложные запросы:

GET /api/sessions/search?attendeeId={id} – все сессии, на которые зарегистрирован участник (демонстрация N+1 и @EntityGraph)

GET /api/sessions/by-event?eventId={id}&speakerName={name} – фильтрация с пагинацией (JPQL + native query)

Bulk-операция:

POST /api/attendees/bulk-register – массовая регистрация участников на мероприятие (список email).

Асинхронные задачи:

POST /api/notifications/send?eventId={id} – запуск рассылки уведомлений (возвращает taskId)

GET /api/notifications/status/{taskId} – статус задачи.

Дополнительные особенности
Обработка ошибок и валидация
Все ошибки возвращаются в едином формате:

json
{
  "timestamp": "2026-03-09T12:00:00",
  "status": 400,
  "error": "Bad Request",
  "message": "Validation failed",
  "path": "/api/events"
}
Валидация осуществляется через аннотации @NotNull, @Size, @Email и др.

Логирование и AOP
Логи сохраняются в файл logs/application.log с ротацией (по 10 МБ, хранятся 30 дней).

Аспект логирует время выполнения каждого метода сервиса (уровень DEBUG).

Кэширование (in-memory индекс)
Для оптимизации повторяющихся запросов с параметрами (например, поиск сессий по дате и спикеру) используется самодельный кэш на основе ConcurrentHashMap. Ключ — объект, содержащий значения параметров запроса (переопределены equals и hashCode). При создании, обновлении или удалении соответствующей сущности кэш инвалидируется.

Асинхронные операции
Рассылка уведомлений выполняется в пуле потоков (настроен через @Async).

Статус задачи хранится в памяти (можно заменить на БД).

Для потокобезопасности счётчика используется AtomicInteger.

Тестирование
Unit-тесты: сервисы изолированно с Mockito.

Интеграционные тесты: проверка работы с БД (H2).

Нагрузочное тестирование: JMeter скрипты в jmeter/.

CI/CD и деплой
Проект использует GitHub Actions для автоматизации:

При push в ветку main запускается сборка Maven и все тесты.

После успешной сборки собирается Docker-образ и публикуется в Docker Hub (или сразу деплоится на PaaS).

Выполняется деплой на Railway (или Render) с использованием переменных окружения.

После деплоя проверяется healthcheck endpoint.

Пример workflow: .github/workflows/deploy.yml

SPA-клиент
Клиентская часть находится в папке /frontend. Написана на React с использованием Axios для запросов к API. Основные страницы:

Список мероприятий с фильтрацией.

Детальный просмотр мероприятия (с сессиями и спикерами).

Управление участниками и билетами.

Админка для массовой регистрации.

Для запуска клиента:

bash
cd frontend
npm install
npm start
По умолчанию клиент обращается к API на http://localhost:8080 (можно настроить прокси).
